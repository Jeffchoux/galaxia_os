# RAPPORT D'AUDIT EXÉCUTIF — Écosystème Galaxia
**Date : 2026-05-31** · Destinataire : Jeff (DG) · Rédacteur : Direction Technique
**Périmètre : 11 audits techniques (avec vérification adverse de tous les findings critiques/élevés) + 5 veilles état-de-l'art**

---

## TL;DR EXÉCUTIF

Galaxia est un **prototype avancé techniquement impressionnant** pour un projet solo de quelques semaines : cockpit SvelteKit complet, orchestrateur Cowork avec sandbox Docker durci, bot Telegram, coder autonome, chaîne d'install signée cosign, et une discipline de coût (« gratuit par défaut ») appliquée partout. Les fondations de sécurité applicative (Argon2id, SQL paramétré, anti-traversée) et l'isolation des sandboxes Cowork sont de **niveau professionnel**.

Mais ce n'est **pas encore un produit industrialisable**. Trois familles de problèmes dominent : (1) une **sécurité d'infrastructure** dangereuse à la racine (sudo NOPASSWD + groupe docker = escalade root triviale depuis n'importe quel agent compromis) ; (2) un **chaînon Hub & Spoke entièrement non opérationnel** (aucune release jamais publiée, clé de signature jamais générée) qui rend la promesse « auto-amélioration » creuse aujourd'hui ; (3) **zéro PME pilote** et un **mélange de marques/domaines** restaurant↔Galaxia qui n'a pas encore touché le terrain.

**Note globale pondérée : 49/100** (détail et pondération en section suivante). C'est la note d'un MVP sain mais loin de la distribution PME.

**Les 3 risques qui empêchent d'industrialiser :**
1. **Escalade root triviale** — `galaxia` a `sudo NOPASSWD:ALL` + appartenance au groupe `docker` ; le worker Telegram (qui exécute Claude en `bypassPermissions`) n'a aucun durcissement systemd. Une injection de prompt = compromission totale du VPS et de toutes les PME co-hébergées.
2. **Hub & Spoke fantôme** — clé cosign jamais générée, registre vide, timer jamais installé, aucune fille jamais mise à jour. Le différenciateur n°2 du produit est un POC non câblé.
3. **Pas de validation marché** — 0 PME pilote, friction d'install rédhibitoire pour la cible « comptable de 54 ans », licence non publiée (repo déjà public sans LICENSE).

**Les 3 quick-wins (impact maximal, effort minimal) :**
1. **Supprimer/restreindre `/etc/sudoers.d/galaxia` + sortir `galaxia` du groupe docker** (ou socket-proxy). 1 ligne, supprime la classe d'escalade la plus grave.
2. **Durcir `galaxia-tg-worker.service`** (`NoNewPrivileges`, `ProtectSystem=strict`, `ProtectHome`) — aligner sur `galaxia-bot.service` qui le fait déjà.
3. **Poser `MAIL_PROVIDER=brevo` + auth Caddy devant nemoclaw + headers de sécurité HTTP** — trois corrections de config qui ferment des trous documentés.

## RISQUES PRIORITAIRES (findings CONFIRMÉS critical/high uniquement)

> Méthode : seuls les findings dont le verdict adverse est `confirmed` ou `partially-confirmed` avec sévérité corrigée **high/critical** sont retenus. Les findings **réfutés ou rétrogradés** sont listés en fin de section pour transparence.

### 1. Escalade root depuis tout agent compromis — `sudo NOPASSWD:ALL` + groupe docker
- **Sévérité confirmée : CRITICAL** (SEC-AGT-02)
- **Fichier :** `/etc/sudoers.d/galaxia:1` (`galaxia ALL=(ALL) NOPASSWD:ALL`), `id galaxia` → groupe `docker`.
- **Impact business :** Tout processus tournant sous `galaxia` (coder, cowork, worker Telegram) qui est manipulé par une injection de prompt peut faire `sudo bash` **ou** `docker run -v /:/host --privileged` → root complet sur le VPS, accès aux clés SSH, à tous les secrets et **aux projets clients co-hébergés**. La vérification a montré que le worker Telegram (sans `NoNewPrivileges`) peut effectivement `sudo -n` sans mot de passe — **ce qui invalide la note mémoire « le bot ne peut pas sudo »**.
- **Action :** Supprimer `/etc/sudoers.d/galaxia` ou le réduire à une allowlist stricte (`systemctl restart galaxia-*`). Sortir `galaxia` du groupe docker **ou** placer un socket-proxy (Tecnativa). Vérifier que cloud-init ne recrée pas le fichier au reboot.
- **Effort : quick-win** (mais le retrait du groupe docker demande un test des agents).

### 2. Worker Telegram sans durcissement, exécute Claude en `bypassPermissions`
- **Sévérité confirmée : HIGH** (SEC-AGT-03 / OPS-12 / PY-TG-01)
- **Fichier :** `ops/galaxia-tg-worker.service` (aucun `NoNewPrivileges`/`ProtectSystem`/`ProtectHome`), `agents/telegram/worker.py:140-143` (`--permission-mode bypassPermissions --max-turns 80`, `cwd=/home/galaxia/galaxia-project`).
- **Impact business :** Claude headless avec écriture totale sur le repo + secrets (`ANTHROPIC_API_KEY`, **`GH_TOKEN`**) dans le même process. Un ordre conduisant Claude à lire du contenu hostile, ou un compte Telegram compromis (le seul garde-fou est `ALLOWED_CHAT_ID`), donne un shell Claude sur le serveur client. Combiné au risque n°1 → root.
- **Action :** Aligner sur `galaxia-bot.service` (`NoNewPrivileges=true`, `ProtectSystem=strict`, `ProtectHome=read-only`, `ReadWritePaths` limités) ; idéalement sandboxer le worker dans le même Docker que Cowork ; retirer `GH_TOKEN` de l'env du worker (il n'en a pas besoin — voir SEC-AGT-08). Ajouter une consigne anti-injection au `SYSTEM_PREAMBLE`.
- **Effort : medium.**

### 3. Conteneur NemoClaw/OpenShell en quasi-root permanent
- **Sévérité corrigée : HIGH** (SEC-AGT-01, rétrogradé de critical car la chaîne d'origine NVIDIA a été vérifiée légitime et aucun socket/FS hôte n'est monté)
- **Preuve :** `docker inspect` → `CapAdd=[SYS_ADMIN,NET_ADMIN,SYS_PTRACE,SYSLOG]`, `apparmor=unconfined`, `User=0`, pas de limite mémoire/pids, réseau NAT sortant, **running depuis 9 jours, unhealthy**.
- **Impact business :** Un composant de sécurité tiers tourne avec les primitives d'évasion de conteneur les plus dangereuses, sans surveillance, sur la mère. `SYS_ADMIN` + `apparmor=unconfined` est une primitive d'évasion connue.
- **Action :** `docker stop`/`rm` jusqu'à décision d'intégration formelle ; si conservé, recréer sans capabilities dangereuses, `--user` non-root, `--read-only`, limites mémoire/pids, AppArmor activé.
- **Effort : quick-win.**

### 4. MCP filesystem write/edit exposé à Claude en mode pro (cockpit)
- **Sévérité confirmée : HIGH** (SEC-CKP-02, aggravé par SEC-CKP-09 injection via documents)
- **Fichier :** `apps/cockpit/src/lib/server/tools.ts:218` (`loadAllTools()` ne filtre rien), `claude.ts:291-303` (mode pro passe tous les outils MCP), serveur filesystem scopé sur tout `/home/galaxia/galaxia-project`.
- **Impact business :** Un utilisateur authentifié (ou une injection de prompt dans un PDF/texte uploadé) peut demander à Claude d'écrire/supprimer n'importe quel fichier du repo (units systemd, code de prod, backdoor). Se réplique sur chaque fille.
- **Action :** Appliquer en mode pro la même allowlist read-only qu'en mode free, ou restreindre l'écriture à `data/`+`briefs/` ; ajouter un wrapper `<document source="user_upload">` signalant le contenu non fiable.
- **Effort : medium.**

### 5. Magic links loggés en clair (provider mail « console » en prod)
- **Sévérité confirmée : HIGH** (SEC-CKP-01)
- **Fichier :** `env.ts:81` (`MAIL_PROVIDER ?? 'console'`), `mail.ts:23` (`console.log(... magic link ...)`). `.env` prod ne contient ni `MAIL_PROVIDER` ni `BREVO_API_KEY`.
- **Impact business :** Tout token de connexion (TTL 15 min) est écrit dans journald → quiconque lit les logs prend la main sur un compte. Sur les filles PME où le magic link sera le mode de connexion principal des clients, c'est une fuite systémique.
- **Action :** Poser `MAIL_PROVIDER=brevo` + clé ; faire échouer `install.sh` si `console` en prod ; stocker le hash SHA-256 du token, pas le token brut (SEC-CKP-08).
- **Effort : quick-win.**

### 6. nemoclaw.galaxia-os.com sans auth au niveau Caddy
- **Sévérité confirmée : HIGH** (SEC-CKP-03)
- **Fichier :** `caddy/Caddyfile:55-61` — aucun `basic_auth`/`forward_auth` ; auth uniquement par fragment URL (jamais transmis au serveur).
- **Impact business :** Dashboard d'un composant de sécurité exposé si le backend ne s'auto-protège pas. La note mémoire indique qu'une porte basic_auth a été posée à l'audit du 31/05 — **à vérifier qu'elle est bien dans le Caddyfile du repo et pas seulement sur le VPS**.
- **Action :** Ajouter `basic_auth` ou `forward_auth` dans le bloc Caddy du repo (ne pas dépendre du seul backend).
- **Effort : quick-win.**

### 7. Absence de rate-limiting sur /login et /api/chat
- **Sévérité corrigée : HIGH** (SEC-CKP-04 confirmé high ; CK-SEC-01 rétrogradé critical→high car Argon2id limite le débit)
- **Fichier :** `login/+page.server.ts:29-38`, `api/chat/+server.ts`, `caddy/Caddyfile` — aucun rate-limit nulle part. Le Caddy ne protège que nemoclaw, pas le cockpit.
- **Impact business :** Bruteforce du mot de passe admin (seule protection du compte de Jeff), et **abus de coût API** : chaque message pro = appel Opus (~0,17 $/msg) ; une boucle vide la clé ou ruine la viabilité du modèle à 10 €/mois.
- **Action :** `rate_limit` Caddy sur `/login` et `/api/*` ; lockout après N échecs ; étendre le cap `COWORK_MAX_USD_PER_TASK` au chat.
- **Effort : medium.**

### 8. Coût d'exécution in-sandbox Cowork non plafonné
- **Sévérité corrigée : HIGH** (AG-01 rétrogradé critical→high : coût borné par timeout/concurrence mais non mesuré)
- **Fichier :** `agents/cowork/orchestrator.mjs:59-65` — `cost_micros` ne suit que planner + synthèse ; les sous-tâches (le gros du travail) appellent le modèle via la CLI hors-process et **ne sont jamais comptées**. Le cap est testé contre une fraction du coût réel.
- **Impact business :** Un manager PME peut dépasser silencieusement son budget API d'un facteur 5-10× sans déclenchement du kill. Risque financier systémique à l'échelle N PME.
- **Action :** Remonter `cost_usd` de la sandbox via la ligne sentinelle JSON, ou utiliser `--output-format json` dans l'entrypoint, et l'additionner au cap.
- **Effort : medium.**

### 9. Fiabilité Cowork : tâches orphelines, DAG bloqué, fuite disque
- **Sévérité confirmée : HIGH** (COW-01, COW-02, COW-03 / AG-06 / OPS-09)
- **Fichiers :** `orchestrator.mjs:831-858` (au redémarrage, les tâches `planning`/`running` ne sont jamais reprises → bloquées à vie) ; `651-661` (une sous-tâche en erreur avec descendantes → la mère reste **à tort** en `awaiting_approval`, en busy-loop) ; `426` (workspaces `~/.local/share/galaxia/cowork` jamais nettoyés → disque plein → SQLite ne peut plus écrire → cockpit tombe).
- **Impact business :** Tâches « en cours pour toujours », faux « en attente de validation », et saturation disque silencieuse sur VPS PME 20 Go. Tickets support incompréhensibles à l'échelle.
- **Action :** Reset au boot (`UPDATE ... WHERE status IN ('planning','running')`) ; passer en `skipped`+`done` les descendantes d'une sous-tâche échouée ; `rm -rf` du run-dir après synthèse + job de nettoyage des orphelins.
- **Effort : quick-win à medium.**

### 10. Backup absent (SQLite + clé privée cosign) & rollback non implémenté
- **Sévérité confirmée : HIGH** (OPS-04) / **rétrogradé MEDIUM** (OPS-05, SC-HUB-06 : POC, aucune fille en prod)
- **Fichiers :** `/opt/galaxia/backups/` créé mais vide, aucun timer ; clé privée cosign sans procédure de sauvegarde ; `galaxia-update.sh:121` applique sans healthcheck ni retour arrière.
- **Impact business :** Un crash disque détruit l'historique conversationnel ; **la perte de la clé privée cosign casse définitivement et irréversiblement la distribution signée vers toutes les filles** (point unique de défaillance sur la racine de confiance).
- **Action :** Timer quotidien `sqlite3 .backup` (rétention 7 j) ; sauvegarde hors-site chiffrée de `cosign.key` ; healthcheck + rollback N-1 avant toute fille en prod.
- **Effort : medium.**

### 11. Dépendance Anthropic totale sur les agents critiques
- **Sévérité confirmée : HIGH** (TECH-01)
- **Fichiers :** coder, cowork et cockpit importent tous le SDK Anthropic ; `getAnthropicKey = required(...)`. Aucun fallback Ollama sur le chemin agentique (Groq = chat nu seulement).
- **Impact business :** Hausse tarifaire, blocage de clé ou panne API Anthropic → arrêt complet des agents cœur de produit. Contradiction directe avec l'argument de vente n°1 (souveraineté). Voir contradiction stratégique ci-dessous.
- **Action :** Couche d'abstraction LLM (LiteLLM) en priorité sur le coder ; documenter un plan de continuité local.
- **Effort : large.**

### Findings RÉTROGRADÉS ou RÉFUTÉS par la vérification adverse (transparence)
- **RÉFUTÉ — `apps/cockpit/.env` root:root bloquerait le service (OPS-01, était critical)** : faux. systemd lit l'`EnvironmentFile` en tant que root **avant** de droper vers `galaxia` ; le service tourne, toutes les clés sont injectées. État voulu et documenté. → info.
- **RÉFUTÉ — Contradiction souveraineté « promesse absolue » (STRAT-01, était critical)** : le doc produit documente honnêtement les deux modes (cloud-avec-clé / 100 % local). Reste un sujet de cadrage marketing → low. (La vraie contradiction est ailleurs : pas de moteur local sur le chemin agentique, voir TECH-01.)
- **RÉFUTÉ — Restaurant : 4 décisions bloquantes (GTM-04, était high)** : 2/4 déjà tranchées+provisionnées le 31/05 (Scaleway TEM, adresse postale), Stripe partiel, base légale sur défaut → low.
- **PARTIELLEMENT RÉFUTÉ — Ollama absent du compose casse les filles (OPS-02, était critical)** : Ollama tourne en systemd hôte par conception (hybride), pas en conteneur. Le défaut réel est un commentaire trompeur → low.
- **RÉTROGRADÉ — DB SQLite world-readable (SEC-AGT-06, high→medium)** : le home en 750 bloque l'accès cross-user ; défaut = absence de défense en profondeur, pas fuite effective.
- **RÉTROGRADÉ — OpenClaw/NemoClaw absents du code (STRAT-02, critical→medium)** : ils existent dans la couche install/deploy/sandbox livrée ; le problème est un décalage de **positionnement** (« moteur agentic » revendiqué ≠ Claude SDK réel), pas du vaporware.
- **RÉTROGRADÉ — clé publique cosign non distribuée / supply-chain (SC-HUB-01/02/04, plusieurs critical→medium/high)** : le client `galaxia-update.sh` est **fail-closed** par défaut (refuse les manifestes non signés), donc pas de RCE silencieuse ; ce sont des gaps de complétude POC, pas des failles exploitables aujourd'hui.
- **RÉTROGRADÉ — `listMessages` sans LIMIT (CK-PERF-01, high→low)** : volumétrie réelle ridicule (72 messages en base) ; dette latente, pas un risque actif.
- **RÉTROGRADÉ — worker restaurant verrou SQLite (PY-WORK-01, high→low)** et **Whisper upload non borné (PY-WHIS-01, high→medium)** : surface locale / WAL+timeout 15s atténuent.

## DÉTAIL PAR DOMAINE

### Sécurité web cockpit (62/100)
**Forces :** Argon2id (`@node-rs/argon2`), comparaison HMAC en temps constant, cookies `httpOnly/secure/SameSite=lax`, SQL 100 % paramétré avec scoping `user_id`, anti-traversée `realpathSync` dans la vue Code, magic links à usage unique atomiques, écoute sur `127.0.0.1:3001` derrière Caddy uniquement.
**Findings durs :** magic link en clair dans logs (#5), MCP write non bridé (#4), nemoclaw sans auth Caddy (#6), pas de rate-limit (#7). **Medium :** pas de headers HTTP (CSP/X-Frame-Options/HSTS), rôle `admin` déclaré mais jamais vérifié dans les routes, session HMAC sans révocation serveur (TTL 30 j), tokens magic link en clair en base, injection prompt via documents, preview HTML sans sandbox iframe.

### Sécurité agents & infrastructure (42/100 — le plus préoccupant)
**Forces :** sandbox Cowork de **niveau production** (`--read-only --cap-drop=ALL --no-new-privileges --pids-limit --memory --user dynamique`), réseau interne Docker sans NAT, egress tinyproxy allowlist stricte (2 hôtes, `FilterDefaultDeny`), `.env` en 600, validation anti-injection du `SUBTASK_ID`, kill-switch Telegram.
**Le problème :** ces excellentes barrières sont **contournables** par la couche hôte (sudo NOPASSWD #1, groupe docker, worker TG nu #2, NemoClaw quasi-root #3). C'est l'illustration parfaite d'une sécurité forte au mauvais niveau. **Medium :** `ANTHROPIC_API_KEY` (clé plateforme) injectée inutilement dans les sandboxes, `GH_TOKEN` partagé au worker TG, `CLAUDE_CLI_VERSION=latest` non reproductible.

### Supply-chain / Hub & Spoke (38/100)
**Forces :** choix cosign v2 air-gapped (pas de Rekor) adapté à l'offline souverain, refus fail-closed des manifestes non signés, round-trip signé testé en CI, install.sh `set -euo pipefail` idempotent, Caddy via APT+GPG.
**Le problème :** **rien n'est en production** (SC-HUB-05, confirmé) — `/opt/galaxia/keys/` inexistant, `/var/www/galaxia-updates/` vide, `registry.galaxia-os.com` NXDOMAIN, timer jamais installé, `galaxia-publish.sh` jamais exécuté. Le bootstrap cosign sans vérification de checksum (SC-HUB-02, high) reste à durcir avant GA.

### Qualité code frontend cockpit (52/100)
**Forces :** auth HMAC + magic links bien faits, anti-traversée, SQL paramétré, markdown sécurisé maison, env lazy avec fail-fast au boot, routeur souverain déterministe (pas d'appel réseau pour choisir le modèle).
**Findings :** monolithe `+page.svelte` de **4 314 lignes** (CK-ARCH-01, high — vrai risque de maintenance/conflits de PR), chemins `/home/galaxia` hardcodés dans `mcp.ts` qui **cassent les MCP sur toute fille Dockerisée** (CK-DIST-01, high), couverture unitaire faible (CK-TEST-01 rétrogradé medium : un smoke Playwright CI existe).

### Qualité code agents Node (52/100)
**Forces :** atomicité SQLite `BEGIN IMMEDIATE` exemplaire, sandboxing durci, gate d'approbation solide, prompts statiques cache-friendly, dédup PR robuste du coder.
**Findings :** trio fiabilité Cowork (#9), double-insertion de plan si crash (COW-04, rétrogradé medium car non déclenché aujourd'hui), fallback silencieux dans `galaxia_read_brief` (MCP-01), GitHub Trending scraping fragile.

### Qualité code Python (71/100 — le mieux noté)
**Forces :** RGPD by design **testé** (suppression list irréversible, token unsub obligatoire, dry-run double-verrouillé), validation anti-hallucination LLM, SQL paramétré, token Telegram protégé des logs.
**Findings :** limite upload Whisper absente (PY-WHIS-01, medium), worker restaurant sans try/except sur le verrou (PY-WORK-01, low), bypassPermissions TG (#2), zéro test bot/Whisper.

### Architecture agentique & IA (62/100)
**Forces :** sandbox exemplaire, gate humain cohérent, discipline « gratuit par défaut », DAG acyclique garanti par Zod, egress filtré, documentation honnête des limites.
**Findings :** coût in-sandbox non plafonné (#8), Groq cloud par défaut (AG-02, contradiction souveraineté), daemon Cowork séquentiel (AG-03, bloque le multi-utilisateur), `llama3.1:8b` daté (AG-04, rétrogradé low car veille = source secondaire à dégradation gracieuse), prompt de synthèse = prompt du planner (AG-05).

### DevOps / redéployabilité (52/100)
**Forces :** Hub & Spoke conçu, durcissement systemd large, CI 8 jobs (shellcheck, e2e Ubuntu, Playwright, round-trip cosign), logrotate, RandomizedDelaySec sur les timers.
**Findings :** pas de healthcheck Docker (OPS-03, medium), backup absent (#10), `:latest` non versionné (OPS-07), `galaxia-cowork-egress.service` sans `[Install]` (OPS-08, ne survit pas au reboot), images sandbox non buildées en CI (OPS-13).

### Stratégie produit (31/100) & Go-to-Market (28/100)
**Forces :** différenciation réelle et défendable (espace blanc « zéro-cloud + zéro-DSI » qu'aucun concurrent ne couvre — voir veille marché), décisions tracées/réversibles, install pensée non-dev, restaurant rigoureux.
**Findings :** 0 PME pilote (GTM-01/STRAT-pivot, high — risque assumé et time-boxé au 21/06), friction install (clé API + BotFather + serveur SSH) vs promesse no-IT (GTM-02, high), licence non publiée sur repo déjà public (GTM-05, high), positionnement marque restaurant↔Galaxia à séparer (GTM-08), canaux d'acquisition non définis (GTM-09).

### RGPD / conformité restaurant (67/100)
**Forces :** privacy-by-design testé, dry-run + send_enabled + ring 0 triple-verrouillés, source OSM/ODbL traçable, audit log survivant aux effacements, rétention 90 j, pas de CB stockée (Stripe opaque).
**Findings (tous rétrogradés high→medium car dry-run, aucun envoi réel) :** source des données absente du corps email (RGPD-REST-01, Art.14), test de mise en balance (LIA) non formalisé (RGPD-REST-02), mentions légales/CGV/politique de confidentialité absentes avant facturation (RGPD-REST-03), endpoint `/u/{token}` non déployé.

## VEILLE & MODERNISATION (état-de-l'art au 2026-05-31)

### Modèle local : OUI, `llama3.1:8b` est à remplacer (quick-win, sans rien casser)
`llama3.1:8b` (juillet 2024) est fonctionnel mais dépassé. **Action immédiate :** `ollama pull qwen3:8b` — surpasse llama3.1 sur code (+3,4 pts HumanEval), reasoning et tool-calling, **même empreinte RAM** (~5,5 Go en Q5_K_M), licence Apache 2.0. Alternative conservatrice drop-in : `llama3.3:8b`. **Passer la quantization de Q4_K_M à Q5_K_M** pour le tool-use (+0,8 Go RAM, fiabilité function-calling nettement meilleure). Pour PME <8 Go RAM : `phi4-mini` (~3 Go). **Garder Ollama** comme moteur (bon choix souverain ; ne migrer vers vLLM/SGLang que si 5+ utilisateurs concurrents avec GPU). **Ne jamais déployer TGI** (maintenance mode depuis déc. 2025).
Sources : `https://www.spheron.network/blog/deepseek-vs-llama-4-vs-qwen3/`, `https://www.promptquorum.com/power-local-llm/best-local-models-tool-calling-2026`

### Frameworks agentiques : garder l'orchestrateur DAG maison, mais réduire la dépendance Anthropic

## CONTRADICTION STRATÉGIQUE À TRANCHER

**La vraie contradiction n'est pas dans le discours marketing** (le doc produit documente honnêtement cloud-vs-local — STRAT-01 réfuté), **mais dans l'architecture** :

> Galaxia se vend « souverain », mais **tous les agents cœur de produit** (coder, Cowork, chat pro) **dépendent exclusivement du SDK Anthropic**, et le mode « gratuit par défaut » route le chat vers **Groq (cloud US)**. Le seul moteur local (Ollama) ne sert que la veille — **il n'existe sur aucun chemin agentique ni sur le chemin de chat par défaut.** Le « zero cloud-call » revendiqué (PRODUCT-VISION) est donc faux pour le comportement par défaut (TECH-01, AG-02, tous deux confirmés).

**À trancher par Jeff :** soit (a) assumer un positionnement « données chez vous, traitement LLM délégable » et faire d'Ollama le **défaut réel du wizard PME**, soit (b) construire la couche d'abstraction LLM (LiteLLM) pour offrir un vrai mode 100 % local sur les agents. Le statu quo expose à un reproche immédiat d'un DPO ou d'un acheteur averti, et à une rupture de service si Anthropic change ses conditions.

Sujet connexe à arbitrer : **OpenClaw/NemoClaw** présentés comme « moteur agentic » / « couche sécurité » alors que le cœur tourne sur Anthropic (STRAT-02). Décider : intégrer réellement, ou réviser BRIEFING/STACK/ARCHITECTURE pour décrire la stack réelle. Préalable à toute présentation partenaire.
